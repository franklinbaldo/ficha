# FICHA — Auditoria e próximos passos (2026-07-18)

**Data:** 2026-07-18 · **HEAD de referência:** `1a27ff22` · **Auditoria anterior:**
[`vision-blockers-2026-07.md`](vision-blockers-2026-07.md) (2026-07-13)

Esta auditoria revisita o projeto cinco dias após o diagnóstico dos bloqueios e,
mais importante, **depois da primeira execução agendada do `etl-monthly` com o
stderr visível** (2026-07-15). O quadro mudou de forma relevante: o bloqueio nº 1
da auditoria anterior — a resolução do mês — **caiu**, e o pipeline avançou até o
penúltimo passo antes de morrer num ponto novo, específico e barato de consertar.

---

## TL;DR — o que mudou desde 2026-07-13

1. **A resolução do mês parou de ser o gargalo.** O cron de 2026-07-15
   (run `29391359479`) resolveu `2026-07`, baixou os 37 ZIPs da RFB, extraiu
   ~26 GB de CSV e carregou as 4 tabelas grandes no DuckDB
   (`empresa` 69M, `estabelecimento` 71M, `simples` 49M, `socio` 28M) — tudo no
   runner free-tier, em ~47 min de wall-time (bem dentro do orçamento de
   350 min). O modo de falha crônico dos crons de maio/junho/julho
   (`list-snapshots` vazio → `Invalid month ''`) **não se repetiu**.

2. **O pipeline chegou a ~90% do E2E — e provou a máquina toda.** Fases 1
   (download+extract), 2 (load DuckDB) e boa parte da 3 (write parquets)
   rodaram em escala real. Foram escritos com sucesso `cnpj_contatos.parquet`
   (1.66 GB), `cnpj_cnaes.parquet` (1.78 GB) e `enderecos.parquet` (0.69 GB).
   A mitigação de OOM (chunk-per-ZIP, deleção agressiva de CSV, `threads=1`)
   **funcionou** — o pico de memória e disco couberam. Isso valida de uma vez a
   dúvida em aberto desde maio ("o bootstrap nunca fechou E2E em escala").

3. **O novo bloqueio nº 1 é uma flag do leitor de CSV do DuckDB — e já está
   corrigido neste PR.** A run morreu ao escrever `cnpjs.parquet` (o caminho
   *chunked*, que relê `estabelecimento` um CSV por vez):

   ```
   error: transform failed: CSV Error on Line: 14310474
    The parallel scanner does not support null_padding in conjunction with
    quoted new lines. Please disable the parallel csv reader with parallel=false
      file = .cache/raw/2026-07/extracted/estabelecimentos/K3241.K03200Y0.D60711.ESTABELE
   ```

   Causa: `_create_table_from_csvs` (`etl/src/ficha_etl/transform.py`) lia com
   `null_padding=true` mas sem `parallel=false`. O scanner paralelo do DuckDB
   *range-splita* um arquivo único por offset de bytes; com `null_padding` ele
   não consegue reconstruir linhas curtas cujo campo contém uma quebra de linha
   entre aspas que cruza a fronteira do split, e aborta. Na Fase 2 a
   `estabelecimento` é lida como **lista de 10 CSVs** (split por arquivo, sem
   corte intra-arquivo) e passou; no caminho chunked da Fase 3 ela é lida **um
   CSV de cada vez** (arquivo único → split intra-arquivo) e falhou. Por ser
   dependente da posição do dado, o bug ficou latente até o primeiro run que
   chegou nessa fase.

   **Correção aplicada:** adicionar `parallel=false` ao `read_csv` de
   `_create_table_from_csvs`. Como `threads=1` já é a norma aqui (freio de
   spill documentado no call-site do PRAGMA), desligar o leitor paralelo **não
   custa throughput** e torna a leitura determinística nos dois caminhos. É
   exatamente o remédio que a própria mensagem de erro do DuckDB prescreve. Os
   25 testes de `test_transform.py` seguem passando.

**Síntese:** o FICHA está a **um run** do seu primeiro snapshot mensal completo e
automático. O ciclo operacional que nunca fechou sozinho parou de falhar por
resolução de mês e parou de falhar por OOM; faltava só esta flag.

### Dimensão 1 — Operação (era o blocker nº 1; agora quase fechado)
- O `2>/dev/null` que engolia a causa-raiz foi removido (fix `2ef1e6cd`), e o
  primeiro run com stderr visível **pagou o investimento**: mostrou que o mês
  resolve e revelou o próximo obstáculo real em vez de mais um `Invalid month`.
- Com o `parallel=false` deste PR, o próximo `etl-monthly` (agendado ou via
  `workflow_dispatch inputs.month=2026-07`) deve produzir os **7 parquets +
  `companies.zip`** e comitar o manifest pelo bot — o teste E2E pendente desde
  maio.

### Dimensão 3 — Histórico (visão central; ainda não iniciada)
- `backfill.yml` e `ficha-etl rebuild-manifest --from-ia` continuam inexistentes.
  Sem eles, o "Rolodex histórico" (README) segue sendo texto. Este é o maior
  bloco de trabalho *de produto* restante e depende do passo 2 abaixo estar
  fechado.

### Dimensão 4 — Frontend (código pronto, dados ausentes — confirmado)
- Manifest empacotado ainda é `2026-04` com só `cnpjs`/`raizes`/`socios`. As
  abas Pessoa/Endereço/CNAE existem no código porém a barra inteira fica
  escondida (`SearchCNPJ.svelte:367`) porque os parquets não estão no manifest.
- **Botão "Contatos" quebrado no build atual**: `EmpresaFicha.svelte:219`
  renderiza sempre que há `db`, mas `cnpj_contatos` nunca é registrado no
  snapshot 2026-04, então o clique dispara erro de query (não 404 literal).
- Camada atômica (`companies.ts` + protobuf `.pb`) segue **completa, testada e
  não conectada** a nenhum componente — só ao teste.
- Busca por nome ainda é `ILIKE '%...%'` sobre `cnpjs.parquet`
  (`SearchCNPJ.svelte:344`) — derrota bloom filter, baixa centenas de MB.
- **Dívidas de código morto (novas evidências):** `raizes.parquet` (2.86 GB) é
  declarado no manifest, exigido pelo schema Zod, mas **nunca consultado** pelo
  frontend; `getSchema`/`registry` (`schemas/registry.ts`) sem callers; schemas
  Zod de dados nunca validados em runtime (só `ManifestSchema`).
- **Bug latente de manifest:** `SnapshotEntrySchema.files` exige `raizes` e
  `lookups` como obrigatórios; um snapshot futuro que *não* publique `raizes`
  (que é justamente o não-usado) reprova no Zod e **apaga o site inteiro**
  (`manifest.ts:28-29` lança). Ver passo 5.

### Dimensão 5/6 — Coerência e cruft
- `ficha-py` cumpre ADR 0017 parcialmente e é importado de verdade pelo ETL
  (`write_lookup_parquets`), mas **não está no PyPI** (o README manda
  `pip install ficha-py`, que hoje falha).
- Sobrou infraestrutura "Jules": 5 workflows `claude-jules-*.yml` + 20 arquivos
  em `etl/scripts/jules/` sem commits desde 2026-05-09. Candidato a remoção
  (reduz superfície de manutenção e confusão).
- Dependabot PR #40 (urllib3 2.6.3→2.7.0, **security**) aberto há >30 dias.

## Caminho crítico recomendado (ordem importa)

1. **[FEITO neste PR] Corrigir o `read_csv` (`parallel=false`).** Desbloqueia a
   escrita de `cnpjs.parquet` — o único passo entre o pipeline e o 1º snapshot
   completo.

2. **Fechar o primeiro ciclo E2E real (o passo que destrava tudo).** Rodar
   `etl-monthly` via `workflow_dispatch inputs.month=2026-07` (ou aguardar o
   próximo cron) e confirmar: 7 parquets + `companies.zip` publicados no IA,
   URLs verificadas (`verify_snapshot_files`), manifest comitado pelo bot em
   `main`. Isso valida upload+publicação de uma vez e **destrava de graça** as
   abas Pessoa/Endereço/CNAE e o botão Contatos no frontend.

3. **Reparar/aposentar o snapshot 2026-04.** Depois do passo 2, apontar
   `manifest.current` para 2026-07 (completo) e redeploy do site. Isso remove
   o snapshot parcial que causa o botão Contatos quebrado e as abas escondidas.
   *Quick win de produto sem escrever frontend.*

4. **Endurecer o manifest contra o bug latente do Zod (barato, alto valor).**
   Antes de multiplicar snapshots, tornar `raizes` opcional em
   `SnapshotEntrySchema.files` (ou parar de publicá-lo, já que o frontend nunca
   o lê) para que um snapshot sem `raizes` não apague o site. 1 linha de schema.

5. **Implementar o backfill (ADR 0016).** `backfill.yml` (matrix) +
   `rebuild-manifest --from-ia`. É o que materializa o "Rolodex histórico" — o
   maior bloco de produto restante. Só faz sentido *depois* dos passos 2-3
   comprovarem o ciclo mensal.

6. **Só então, features de visão no frontend** (todas dependem dos dados acima):
   seletor de mês / UI temporal (o schema já suporta N snapshots; falta a UI),
   ligar a camada atômica (`companies.ts`), autocomplete e busca por nome que
   não seja `ILIKE`, rota `/empresa/[cnpj]` para deep-link.

7. **Higiene contínua (paralelizável, baixo risco):**
   - Publicar `ficha-py` no PyPI **ou** ajustar o README para instalação via
     git (hoje o comando documentado falha).
   - Remover a infra "Jules" morta (5 workflows + `etl/scripts/jules/`) se não
     houver plano de uso.
   - Mergear/rebaser o Dependabot #40 (urllib3 security).
   - Remover código morto do frontend (`getSchema`/`registry`; decidir o destino
     de `raizes.parquet`).

**Uma frase:** cinco dias atrás o FICHA estava travado no *começo* do ciclo
operacional; hoje está travado no *fim* — e o último obstáculo era uma flag do
leitor de CSV, agora corrigida. O próximo run mensal é o marco que converte
"código adiantado, dados atrasados" em "primeiro snapshot completo e automático".
