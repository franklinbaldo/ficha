# Hardening — Playwright e2e tests for analytical parquets

## Context

Recent merges to `claude/ficha-perf-plan-v2` shipped four new
analytical parquets exposed via DuckDB-WASM in the browser:
- `cnpjs.parquet` (existing, denormalized)
- `cnpj_cnaes.parquet` (W11) — reverse CNAE lookup
- `cnpj_contatos.parquet` (W12) — reverse contact lookup
- `lookups/<kind>.parquet` (W10) — six per-lookup parquets +
  `attachLookups()` glue

Today `web/` has unit tests (Vitest) but no end-to-end browser
test that exercises the actual DuckDB-WASM path. Without it, a
schema drift or `attachX` regression won't be caught until
production.

## Scope

Add Playwright e2e tests under `web/tests/e2e/` covering:

1. **Smoke**: SearchCNPJ component loads, attaches `cnpjs.parquet`,
   exact-CNPJ lookup returns at least one row from a fixture
   snapshot. (Use the existing `manifest.test.ts` fixture URL or
   commit a small inline parquet fixture.)

2. **Length-14 branch (W4.1)**: typing `00.000.000/0001-00`
   triggers `WHERE cnpj = ?`; typing `acme` triggers
   `WHERE razao_social ILIKE ?`. Assert at the network layer
   (Playwright `page.on('request')`) that distinct query plans
   fire.

3. **Lookup attach (W10)**: `attachLookups()` registers six
   files; running
   `SELECT codigo, descricao FROM lookup_municipios LIMIT 5`
   returns the expected fixture rows.

4. **Reverse CNAE lookup (W11)**: query
   `SELECT cnpj_base FROM cnpj_cnaes WHERE cnae_codigo = ? AND posicao = 0`
   returns expected rows.

5. **Reverse contact lookup (W12)**: query
   `SELECT cnpj FROM cnpj_contatos WHERE valor = ?` returns
   expected rows for a fixture phone/email.

## Test infrastructure

Reuse what's already in `web/`:
- `package.json` likely has Astro / Vitest scripts. Add
  `playwright` + `@playwright/test` to devDeps if missing
  (project uses bun lockfile — `bun add -d`).
- New script `test:e2e` running `playwright test`.
- Fixture parquets: small synthetic ones generated by a
  one-off script, committed under `web/tests/fixtures/`.

For the fixture generator: a simple Python script
`web/tests/fixtures/build.py` using `duckdb` to produce all five
parquets with 10 rows each. Run once locally; commit the .parquet
files (small, ~few KB).

## CI

Add a `web-e2e` job in `.github/workflows/ci.yml` that runs after
`web/` job and executes the e2e suite. Don't make it block the
existing `web/` check; add it as a separate parallel job. Use
`actions/setup-node@v4` and `oven-sh/setup-bun@v2` matching the
existing setup.

## Acceptance criteria

- `bun run test:e2e` (in `web/`) passes locally with 5 e2e tests.
- CI job `web-e2e` runs on PR and is green.
- Fixture parquet files are < 10 KB total (small enough to commit).
- A schema drift in any analytical parquet (e.g., dropping a
  column) makes the corresponding test fail clearly.
